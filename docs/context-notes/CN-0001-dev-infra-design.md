# CN-0001: 개발 인프라 시스템 설계

- 작성일: 2026-03-29
- 단계: Phase 0 (개발 환경 구축)

## 결정 사항

Claude Code 에이전트 + 훅 시스템 + 스킬북 + 문서 체계를 다음과 같이 구성했다:

```
기획서(설계도) — docs/plan/           "뭘 만들어야 하나?"
맥락노트(시방서) — docs/context-notes/ "왜 이렇게 했나?"
체크리스트(공정표) — docs/checklist.md  "어디까지 했나?"
```

## 배경/맥락

- 1인 개발 프로젝트로, 하드웨어(회로/PCB)부터 AI까지 영역이 넓음
- 작업 중간에 며칠씩 비울 수 있어서, "어디까지 했더라?"를 빠르게 파악해야 함
- 도메인이 다양해서(임베디드, 신호처리, ML, 통신) 맥락 전환 시 가이드가 필요함

## 검토한 대안

### 대안 1: 단일 거대 에이전트 프롬프트
- 모든 가이드를 에이전트 파일에 넣기
- **기각 사유**: 토큰 낭비. ESP32 작업할 때 ML 가이드까지 로드할 필요 없음

### 대안 2: 에이전트별 완전 독립 매뉴얼
- 각 에이전트가 자기 도메인 매뉴얼을 전부 내장
- **기각 사유**: 중복이 많음. ch01(ESP32)은 코드 에이전트와 테스트 에이전트 모두 필요

### 대안 3 (채택): 스킬북 챕터 시스템
- 에이전트는 핵심 원칙만 보유, 상세 가이드는 공유 챕터로 분리
- 훅이 자동으로 관련 챕터를 추천 → 에이전트가 Read로 필요한 것만 로드
- **채택 사유**: 중복 제거, on-demand 로딩, 챕터 단위 유지보수 가능

## 구현 내용

### 1. 에이전트 (7개)
| 에이전트 | 역할 | 발동 조건 |
|---------|------|----------|
| senior-code-architect | 코드 작성/리뷰/리팩토링 | 코드 관련 키워드 |
| test-diagnose-runner | 테스트/디버깅/진단 | 테스트/에러 키워드 |
| ekg-project-planner | 기획/문서/보고 | 계획/문서 키워드 |
| hardware-circuit-reviewer | 회로 검토/전원부 설계/PCB 가이드 | 회로/PCB/하드웨어 키워드 |
| signal-data-engineer | 신호처리/필터 설계/데이터 파이프라인 | 필터/신호/ECG/DSP 키워드 |
| ml-model-trainer | ML 모델 설계/학습/경량화/배포 | AI/ML/학습/모델 키워드 |
| reference-researcher | 레퍼런스 분석/데이터시트/부품 조사 | 레퍼런스/조사/데이터시트 키워드 |

### 2. 훅 시스템 (3개)
| 훅 | 타이밍 | 역할 |
|----|--------|------|
| prompt-analyzer | UserPromptSubmit | 프롬프트 분석 → 에이전트+챕터 추천 |
| worklog-writer | UserPromptSubmit | 매 요청마다 일일 작업일지 자동 기록 |
| code-review-reminder | PostToolUse | 코드 위험/에러/보안 귀띔 + 맥락노트/체크리스트 리마인더 |

### 3. 스킬북 (10챕터)
ch01~ch10, 도메인별 상세 가이드. 에이전트가 필요할 때만 Read.

### 4. 문서 체계
- 기획서: 전체 로드맵, 목표, 범위 (아직 정식 작성 전)
- 맥락노트: 의사결정 근거 기록 (이 파일이 첫 번째)
- 체크리스트: 7Phase 기반 공정표 (Phase 0 인프라 완료)

## 근거

- **건설 비유**: 기획서=설계도, 맥락노트=시방서, 체크리스트=공정표
  - 설계도만 있으면 "왜 이 자재?"를 모르고 (→ 시방서 필요)
  - 시방서만 있으면 "얼마나 진행됐나?"를 모름 (→ 공정표 필요)
  - 세 문서가 서로 보완해야 프로젝트 추적이 완전해짐
- **1인 개발**: 머릿속 맥락이 유일한 단일 장애점(SPOF). 문서화로 리스크 분산.

## 주의사항

- 맥락노트를 "모든 변경마다" 쓰면 피로해짐 → **의미 있는 결정**에만 작성
- 체크리스트가 너무 세분화되면 관리 비용 증가 → Phase 단위 + Gate 판정으로 단순화
- 스킬북 챕터는 프로젝트 진행에 따라 실제 경험 기반으로 업데이트해야 함
